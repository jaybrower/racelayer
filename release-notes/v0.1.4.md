# v0.1.4

_Unreleased_

## Added

- **Relative: CarLeftRight side indicator.** The player row now shows small amber chevrons flanking the car number (`◀ #42 ▶`) when a car is alongside on either side, driven by iRacing's `CarLeftRight` proximity field. Suppressed in pit and toggleable per session type under Settings → Relative → "Side Indicator" (default on for race, off elsewhere). Closes #9.
- **Radar overlay re-enabled** in Settings → Radar so the corrected `CarLeftRight` side-edge highlights can be exercised. Default-disabled for all session types so the window stays empty/hidden until opted in.

## Changed

- (none yet)

## Fixed

- **Radar: corrected `CarLeftRight` enum mapping** to match the canonical iRacing SDK values (`0=Off`, `1=Clear`, `2=Left`, `3=Right`, `4=Both`, `5=2Left`, `6=2Right`). The previous mapping was off by one, so in a real session the side-edge highlight read "clear" as "car on left" and shifted everything down a slot. Mock telemetry and the shared type comment were updated to match.
- **Radar: cars in front / behind now actually appear.** The gap strip was reading `CarIdxF2Time` directly, but that field is leader-relative (and 0 for cars without a lap time), so opponents within ±1s of the player never showed up. Switched to the same `lapDistPct` + reference-lap-time math the Relative overlay uses, so the strip is now player-relative and works in practice too. Suppressed while the player is in pit (track-position math is meaningless against pit-lane `lapDistPct`).
- **Radar: opponent box no longer overlaps player when they're nearly alongside.** When an opponent is within ~CAR_H of the player but iRacing hasn't flagged them as side-by-side yet, their centre-lane box now clamps to flush-touching above or below the player box instead of drawing on top of it. Side-lane (left/right) positioning is unchanged.

## Internal

- **Vitest test framework wired up.** New `npm test` / `npm run test:watch` scripts run Vitest against `tests/**/*.test.ts`. Scope is pure logic only — no Electron, no React rendering, no jsdom. Manual scenarios live in `docs/test-plan.md` (added separately).
- **`Test` GitHub Action** runs `npm test` on every PR and on pushes to `main` / `release/v*`, so a broken pure-logic change can't merge silently.
- **Pure logic extracted into `lib.ts` modules** alongside each overlay's component (`PitStrategy/lib.ts`, `Relative/lib.ts`, `Gauges/lib.ts`). The component file now imports the formatter / regression / stint-detection / fuel-stat functions instead of defining them inline — same behaviour, but the logic is now unit-testable without bringing React along.
- **Initial test coverage** (85 unit tests):
  - Stint detection, tire-deg trend window, stint-best delta, fuel-stat fallbacks
  - Closing-rate least-squares regression (samples, time span, outlier dampening, sign conventions for car-ahead vs car-behind)
  - Relative-gap track-position wrap, reference-lap-time fallback chain, iR-change Elo approximation
  - Lap-time / delta / gap / fuel formatters
  - `mergeWithDefaults` forward-compat (old configs, missing fields, malformed branches)
- **Manual test plan** (`docs/test-plan.md`) covers everything Vitest can't: app lifecycle, cockpit-only rendering, connection lifecycle, drag/edit mode, per-overlay scenarios (including pit mode and closing-rate edge cases), Settings, in-app updater, settings persistence, and a 5-minute pre-release smoke loop.
- **Pull-request template** (`.github/pull_request_template.md`) reminds authors to run tests, reference an issue, update release notes, and update the test plan when user-visible behaviour changes.
