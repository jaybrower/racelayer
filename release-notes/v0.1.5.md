# v0.1.5

_Unreleased_

## Added

- **Perf HUD (dev / support-debug tool).** Press <kbd>Ctrl</kbd>+<kbd>Shift</kbd>+<kbd>Alt</kbd>+<kbd>P</kbd> to toggle a small always-on-top HUD that surfaces per-overlay React render times (p50 / p95 / max over a rolling 60 s window) plus whole-app and per-process CPU% / memory from `app.getAppMetrics()`. Off by default; zero overhead when hidden (no samples collected, no IPC traffic, no timers). The shortcut is intentionally not exposed in Settings → Shortcuts — it's a developer / support tool, not a glanceable race feature. See `docs/performance.md` for how to use it. Part of #32.

- **Settings: sidebar-tabbed layout.** The single long scrollable Settings page is now a sidebar nav (left) plus a scrollable pane (right), with six panes:
  - **General** — Launch on startup, Auto-hide unsupported overlays, Overlay Positions reset.
  - **Overlays** — per-session-type matrix grouped by overlay, with a sticky Practice / Qualifying / Race column header that never scrolls out of view, and a one-liner description above each overlay group.
  - **Shortcuts** — Layout Mode and Open Settings keybindings (unchanged behaviour, dedicated pane).
  - **Preview Mode** — enable + session type, with a description that explains what the feature is for.
  - **Updates** — current version + update lifecycle (unchanged).
  - **About** — version, links to the GitHub repo / changelog / new-issue form / MIT license.
  Closes #30.
- **Tooltips on overlay toggles.** Every row in the Overlays matrix has a native `title` tooltip with a one-line description of what the toggle does (e.g. *Side Indicator: amber chevrons next to your car number whenever iRacing flags a car alongside you*). Copy is centralised in `Settings/lib.ts` so future edits don't require chasing JSX.
- **Settings → About — external links open in the default browser** via a new `app:openExternal` IPC. Main-process side validates the URL is http(s) before calling `shell.openExternal`, so a compromised renderer can't fire arbitrary URL handlers.
- **Settings window remembers its size and position** across launches, keyed by monitor configuration — same pattern overlays already use. Resize / move once, and that geometry comes back on every relaunch until you change it. Falls back to defaults on first launch and after a monitor-configuration change. Closes #37.
- **Settings: thin themed scrollbars** in place of the chunky OS-default scrollbar. The sidebar and content pane both use an 8px-wide scrollbar with a transparent track and a barely-visible thumb that brightens on hover — fits the rest of the dark UI instead of fighting it.
- **Settings → Overlays: custom-styled checkboxes** matching the app's blue accent — rounded 16px boxes with a hand-tuned SVG check, hover glow, and focus-visible outline. Native `<input type="checkbox">` is still in the DOM (so keyboard / screen-reader / form semantics all keep working), but is visually replaced by an adjacent `<span>` driven off the `:checked` pseudo-class.
- **Gauges: RPM bar peripherally readable + flashes at shift point.** The bar's fill now paints a single colour reflecting which RPM zone the player is currently in (cyan → green → yellow → red), so the colour is recognisable from peripheral vision rather than only the rightmost slice. At the shift point the bar flashes red ↔ fuchsia at ~6 Hz via pure CSS keyframes — no JS animation tick fighting the telemetry cadence. Closes #36.
- **Gauges: configurable shift-point source.** New "RPM Bar" section under Settings → Overlays exposes:
  - **Source** (radio): "iRacing SDK (with percentage fallback)" (default) — uses iRacing's per-car `ShiftIndicatorPct` field when the car exposes it; falls back to the percentage when it doesn't. "Percentage only" — always trigger from the percentage threshold below regardless of SDK.
  - **Flash threshold** (number input, 50–100% of redline, default 97%) — used directly in percentage mode, used as the fallback in SDK mode.

## Changed

- **"Dev Mode" is now "Preview Mode"** throughout the app. The feature exists so users can preview and configure overlays without an active iRacing connection — that's not development work, and the old name implied otherwise. User-visible surfaces updated: tray-menu checkbox ("Preview Mode") and session radios, Settings → Preview Mode pane heading, status badge, toggle copy, and the in-Settings session-type label ("Preview Session Type"). No behaviour change. Part of #30.

## Fixed

- **Edit mode: overlays no longer grow while being dragged.** The custom drag handler used `BrowserWindow.setPosition()` only, which on Windows with DPI scaling let the OS reapply size constraints between frames and creep the window a pixel or two larger per frame. Width/height are now captured at drag-start and re-asserted on every move via `setBounds({x, y, width, height})`, so the size is pinned for the duration of the drag and the window ends at the exact dimensions it started. Closes #29.
- **Tray ↔ Settings sync for Preview Mode** (and other state-change broadcasts). The main-process `broadcastToAll` helper only iterated the overlay windows and silently skipped the Settings window, so toggling Preview Mode from the tray (or any future tray-driven state) never reached the Settings UI. The state would stay stale until the user closed and reopened Settings. Fan-out now includes the Settings window for state-change channels (`previewMode:changed`, `config:changed`, `update:status`), while high-frequency overlay-only events (`telemetry:update` at ~10 Hz, `overlay:editMode`) route through a separate `broadcastToOverlays` helper to avoid wasted IPC traffic. Pre-existing bug — also affected the old `devMode:changed`.
- **Pit Strategy: stint history no longer carries across session transitions.** Practice / qualifying laps used to bleed into a subsequent race stint — the overlay would show pre-race lap numbers / times indefinitely, and the in-progress race laps wouldn't get recorded at all. Root cause: the lap-history `useEffect` early-returned on `lapLastLapTime <= 0` *before* checking whether the session had reset, and on a `qualifying → race` transition `lapLastLapTime` is briefly 0 in the exact tick where the lap counter drops, so the reset branch never fired. Fixed by extracting the lap-tracking state machine into a pure reducer (`reducePitTracker` in `PitStrategy/lib.ts`) that detects session resets on either (a) `sessionType` moving between two known values or (b) the lap counter going backwards, and runs that detection *before* the rest of the gating. Same fix covers the parallel fuel-sampling effect. Closes #39.

## Internal

- **IPC rename**: `window:getPosition` / `window:setPosition` → `window:getBounds` / `window:setBounds`, and matching preload helpers `getWindowBounds` / `setWindowBounds`. Bounds now include `width` and `height` so the renderer can lock size during a drag. No external API surface changed.
- **Preview Mode internal refactor** alongside the user-visible rename: `src/main/devMode.ts` → `previewMode.ts`; `getDevMode` / `setDevMode` → `getPreviewMode` / `setPreviewMode`; `DevModeState` → `PreviewModeState`; IPC channels `devMode:get` / `devMode:set` / `devMode:changed` → `previewMode:*`; preload helpers `getDevMode` / `setDevMode` / `onDevModeChanged` → `getPreviewMode` / `setPreviewMode` / `onPreviewModeChanged`. No migration needed — preview-mode state was already in-memory only and resets to off on every launch.
- **Settings file split**: the 683-line `Settings/index.tsx` is now a thin layout/state-holder; each pane lives under `Settings/panes/` (`GeneralPane.tsx`, `OverlaysPane.tsx`, `ShortcutsPane.tsx`, `PreviewModePane.tsx`, `UpdatesPane.tsx`, `AboutPane.tsx`), and pure helpers (`keyEventToAccelerator`, `formatAccelerator`, tooltip copy maps) moved to `Settings/lib.ts`. No behaviour change.
- **Overlays matrix: `<table>` → CSS Grid.** The per-session-type matrix was an HTML table, which made spacing between overlay groups frustrating (`<tr>` is render-anonymous so `margin` is silently dropped, and `border-collapse: separate` workarounds for the sticky `<thead>` introduced their own border-doubling traps). Rewrote as a flex column where each row is its own 4-column grid (`1fr 72px 72px 72px`). Columns stay aligned, the sticky column header is just `position: sticky` on a div, and group-header spacing is plain block-level `margin-top`. Eliminates a whole category of table-layout edge cases.
- **Telemetry**: new `shiftIndicatorPct: number` field on `IRacingTelemetry` (`NaN` when the current SDK build or car doesn't expose it). Real-SDK side checks the var map directly before reading so a missing field surfaces NaN instead of a misleading 0; mock side simulates a 0→1 ramp keyed off RPM so preview mode flashes correctly.
- **Config schema**: new `gauges.shiftPoint: { source: 'sdk' | 'percent'; flashThresholdPct: number }`. `mergeWithDefaults` validates the source enum, requires the threshold to be a finite number in `[0.5, 1.0]`, and falls back to defaults on any other value. Forward-compat covered by new unit tests.
- **Pure logic**: new `rpmZone()` in `Gauges/lib.ts` (and matching `RpmZone` type + `RPM_ZONE_BREAKS` constants). Stepped colour mapping; flash trigger respects the configured source with NaN-safe SDK fallback. 19 new unit tests cover every zone boundary, both source modes, the SDK-NaN fallback path, and corner cases.
- **Pit Strategy: lap-tracking state machine extracted to `lib.ts`.** The two `useEffect`s + five `useRef`s that drove lap-history and fuel-sampling are now one `PitTrackerState` ref piped through a pure `reducePitTracker(state, tick)` function. The component shrinks to a single effect; the reducer handles session-transition detection, sticky pit-affected flag, lap-boundary push, and fuel sampling. 18 new unit tests cover the #39 regression, session transitions (both directions), backward-lap rewind, pit-flag stickiness, lap/fuel buffer caps, and bootstrap behaviour.
- **Perf instrumentation** (new `src/main/perfMetrics.ts`, `src/main/perfHud.ts`, `src/renderer/src/hooks/usePerfProfiler.tsx`, `src/renderer/src/pages/PerfHud/`): each overlay route is wrapped in a `<PerfProfiler id="...">` that batches React-Profiler `actualDuration` samples and flushes via `perf:recordRender` IPC at 2 Hz when collection is enabled. Main keeps a ring buffer per overlay (600 samples ≈ 60 s at 10 Hz), polls `app.getAppMetrics()` at 1 Hz, and broadcasts `perf:snapshot` to the Perf HUD window. Disabled by default — `<Profiler onRender>` short-circuits on a single boolean check, no IPC traffic, no timers. New `docs/performance.md` documents the measurement workflow and holds the scenario tables for #32's baseline pass.
- **Manual test plan** (`docs/test-plan.md` → Settings window) rewritten for the six-pane structure, with explicit checks for sticky-header behaviour, tooltip presence, About-pane external-link routing, and main-process scheme validation.
- **Manual test plan** (`docs/test-plan.md` → Drag / layout mode) gained two explicit size-identity checks: full-monitor drag across 100% / 125% / 150% / 200% Windows scaling, and a cross-monitor drag on a mixed-scaling multi-monitor setup.
