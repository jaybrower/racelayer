## Internal

- **`/release` slash command + `CLAUDE.md` branch policy: step 12 reworded from "manually close `ready-to-release` issues" to "verify they auto-closed; close stragglers if any."** In practice, GitHub auto-closes issues whose `Closes #N` keywords appear in commit messages that land on the default branch — feature PRs are squash-merged into the release branch (preserving the keyword in the squashed commit message), and the release branch merges into main as a merge commit (per step 6), so those individual commits arrive on `main` and trigger auto-close. Confirmed on the v0.1.6 release: all 6 milestone issues were already `[CLOSED]` by the time step 12 ran. Closes #56.
- **`/release` skill — mechanical steps wrapped as `npm run release:*` scripts.** Four new Node scripts under `scripts/release/*.mjs` (`finalize`, `promote`, `publish`, `open-next`) cover the repeatable command clusters of the pipeline — branch + edit + commit + PR + CI wait + merge for finalize and promote, `npm run dist` + artifact verify + GitHub Release + uploads for publish, branch + bump + scaffold + milestone for open-next. The skill is reduced to walking the four manual gates (test pass, scope confirm, build readiness, next-version pick) and invoking the scripts between them. All four scripts support `--dry-run` for validating preconditions without mutating, and exit non-zero on CI/precondition failure so the skill surfaces problems instead of swallowing them. Closes #59.
