# v0.1.6

_Released 2026-05-16_

## Added

- **Global overlay scale.** Settings → General now has a four-preset picker (75% / 100% / 125% / 150%) that scales every overlay up or down for different monitor sizes and resolutions. Applied via `webContents.setZoomFactor` so text, SVG, and pixel-perfect borders all scale crisply, and each overlay's window dimensions are multiplied by the new/old ratio at the same time so content doesn't overflow at higher scales or leave dead space at lower scales. Persisted in `global.overlayScale` config so the choice survives restarts. Per-overlay scale + continuous-slider variant deferred to a later release. Closes #14.
- **Pit Strategy: race-endpoint awareness + countdown + urgency colour ramp.** The Pit Window section now knows when the race ends and adjusts its display accordingly. Three render modes (closes #12):
  - **Finish on fuel** — when fuel outlasts the remaining race laps, the row shows a green "✓ Finish on fuel" with "N laps left" on the right. No more confusing "Pit by Lap 47" messages when the race ends on lap 30.
  - **Pit by Lap X (in N laps)** — the fuel-forced-pit case picks up an "in N laps" countdown so the player can read urgency at a glance without subtracting.
  - **Urgency colour ramp** — the row recolours as the forced pit approaches: green ("safe") when 6+ laps away, amber ("warn") at 3–5 laps, red ("danger") at ≤ 2 laps.
  Driven by a new `sessionLapsRemain` telemetry field plumbed through from iRacing's `SessionLapsRemain`. Defensive guards treat sentinel values (-1 / 32767 in timed races) as "no info" and fall back to the previous "Pit by Lap N" behaviour. Preview Mode simulates a 30-lap race so the new logic exercises end-to-end without iRacing connected.
- **Perf HUD: Debug panel for log-level controls.** The secret <kbd>Ctrl</kbd>+<kbd>Shift</kbd>+<kbd>Alt</kbd>+<kbd>P</kbd> overlay grows a Debug section showing the currently-applied log level as a segmented control (`error` / `warn` / `info` / `debug`). Changes take effect immediately and persist across restarts in `userData/log-level.json`. The panel also shows the build-tier default ("Default: warn (stable build)") and a Reset button that clears the override. A "Reveal logs in Explorer" button opens the log folder for one-click sharing during support sessions. Surfaced only in the Perf HUD — not in the regular Settings UI — so it stays out of end users' way. Closes #50.

## Changed

- **Log-level defaults are now build-tier-aware** (#50). Stable distribution builds default to `warn` — no more chatty info-level lifecycle messages bloating end users' `%APPDATA%/Roaming/racelayer/logs/main.log`. Prerelease / `dist:pre` test builds default to `info` (verbose enough for development iteration). Dev runs (`npm run dev`) default to `debug` (full firehose, including electron-updater internals). The previous hardcoded `debug` level — added during the #46 investigation — moves to a build-tier-conditional default.
- **Updates pane support footer** ("Updater logs: …") now only appears when the current log level is `info` or `debug`. On a stable default install (level `warn`), the file has nothing useful for support to read, so the footer is suppressed. When a dev / support session bumps the level via the Perf HUD's Debug panel, the footer reappears.

## Fixed

- **Settings → Updates: "Check for updates" button no longer silently does nothing.** Previously, clicking the button could result in zero UI feedback in some paths — no `Checking…`, no error, no `Up to date` — because the renderer's state transitions were entirely event-driven from `autoUpdater`, and `autoUpdater` can silently resolve without firing any lifecycle events (notably in unpackaged dev runs, and via certain feed-fetch failures the library swallows). The button now (a) emits `Checking…` synchronously on click before calling `autoUpdater`, (b) has a 30 s safety-net timeout that surfaces an error if no follow-up event arrives, (c) detects the resolve-with-null silent path and surfaces a clear "Update server returned no version info" error, (d) shows the underlying error message under the row instead of just "Update failed", (e) renders a distinct "Running from source" state for unpackaged builds. Additionally, the update channel is pinned to `latest` AND `allowPrerelease` is forced to `false` so `dist:pre` test builds (versions like `0.1.0-autoUpdateTest.1`) check the stable channel correctly. Both pins are needed: without the `allowPrerelease` override, electron-updater auto-flips it to `true` whenever the installed build has a prerelease tag, which routes `GitHubProvider.getLatestVersion()` into a channel-filtering branch that only matches releases whose entry channel equals the current channel or is `null`/`alpha`/`beta` — stable releases (with no channel tag) can't match a custom channel like `latest`, producing the confusing "No published versions on GitHub" error against a feed that demonstrably contains the latest release. Confirmed by tracing the live debug logs through the electron-updater 6.8.3 source. Closes #46.

## Internal

- **`/release vX.Y.Z` slash command** at `.claude/commands/release.md` walks the full release pipeline end-to-end: finalize PR (release-notes `_Unreleased_` → `_Released_` + version-bump sanity), `release/* → main` merge, tag + push, `ready-to-release` milestone close-out, `npm run dist` build, GitHub Release creation with the `## Internal` section stripped from the user-facing body, artifact upload, and opening the next release branch + milestone. Documented inline with stop-and-ask gates around irreversible steps. Closes #44.
- **`npm run dist:pre` accepts a full version string**, not just a suffix. Detection by first character: digit → full semver-ish version used verbatim (e.g. `npm run dist:pre -- 0.1.0-autoUpdateTest.1`); letter → suffix appended to package.json version (e.g. `npm run dist:pre -- beta.1`, existing behaviour preserved). Surfaced while debugging #46 — needed a build *older* than the latest published release to test the update flow without temporarily editing `package.json`. Closes #48.
- **`electron-log` wired as `autoUpdater.logger`.** Full updater lifecycle writes to `%APPDATA%/RaceLayer/logs/main.log` on installed builds, where there's no terminal to read console output from. Path also surfaced in the Updates pane footer as a support-debug affordance (conditional on log verbosity per #50). Part of #46.
- **Logging system moved to its own module** (`src/main/logging.ts`). Owns `electron-log` initialisation, build-tier-aware default-level policy (dev → debug, prerelease → info, stable → warn), user override persistence at `userData/log-level.json`, and the broadcast that keeps renderer-side UI in sync. `updater.ts` no longer owns any logging config — it just borrows a scoped logger. `broadcastToAll` now reaches the Perf HUD window too (added so `log:level-changed` can flow to the Debug panel; benign for every other channel since the HUD ignores them). Part of #50.
- **Updater diagnostic probes.** On every `checkForUpdates` invocation, the updater module now logs a pre-flight state snapshot (`feedURL`, `channel`, `allowPrerelease`, `allowDowngrade`, `autoDownload`, current app version, packaged-status) before calling `autoUpdater.checkForUpdates()`. On rejection, a parallel raw-`https` probe of the Atom feed and `/releases/latest` URLs runs and logs the response status, redirect Location, content-type, and first 512 bytes of the body — disambiguates "electron-updater HTTP-layer bug" from "host can't reach GitHub" / "GitHub returning unexpected data". Part of #46.
- **`dev-app-update.yml`** committed at repo root so a future debugging session can toggle off the `state: 'dev'` short-circuit in `src/main/updater.ts` and exercise the real GitHub feed in dev mode without setup. Part of #46.
- **Telemetry pipeline: `sessionLapsRemain: number`** added to `IRacingTelemetry`. Read from iRacing's `SessionLapsRemain` SDK field via `varMap.has()` guard (returns `-1` when missing on this SDK build, mirroring the `shiftIndicatorPct` pattern). Mock telemetry simulates a 30-lap race by decrementing from `30 - currentLap`. Consumed by `PitStrategy/lib.ts → computeFuelStats()` for race-endpoint awareness (see #12). Part of #12.
- **`src/main/overlayScale.ts`** new module owning the global scale state: validation against the `OVERLAY_SCALE_OPTIONS` preset list, `applyScaleToWindow` (sets `webContents.setZoomFactor` after `did-finish-load` so the zoom factor survives renderer reloads), and `handleScaleChange` (multiplies every overlay window's bounds by the new/old ratio + re-applies zoom). `registerConfigHandlers` grew an optional `onSet` callback so main can react to specific config changes without piggybacking on the renderer-side broadcast. 5 new unit tests cover the `global.overlayScale` mergeWithDefaults path: forward-compat default, all four presets preserved, hand-edited bogus values (out-of-set numbers, NaN, strings, null) all fall back to 1.0. Part of #14.
- **Pit Strategy lib gains race-endpoint logic + 10 new unit tests.** New `urgencyFor(lapsUntilPit, finishOnFuel): PitUrgency` extracted as a pure helper (own boundary table; tested independently). `computeFuelStats` returns new fields: `lapsUntilPit`, `finishOnFuel`, `lapsLeftInRace`, `urgency`. Sentinel handling for timed-race `sessionLapsRemain` values (-1, 32767, fractional) covered by explicit tests. `URGENCY_DANGER_THRESHOLD` / `URGENCY_WARN_THRESHOLD` / `MAX_USABLE_LAPS_REMAIN` exported so the test suite + future UI tweaks aren't dependent on magic numbers. Part of #12.
